# Default Stack

Codex の標準機能を優先して改善ループを作るときは、次の順で組む。

## Layer 1: AGENTS.md

保存するもの:

- 目的
- 停止条件
- 承認基準
- レポート形式
- 毎回守るベストプラクティス

向いているもの:

- 長く残る原則
- repo 全体に効かせたい判断基準

## Layer 2: skill

保存するもの:

- 繰り返し呼ぶ実行手順
- インタビュー手順
- 出力契約

向いているもの:

- 自然文 trigger
- 半構造化ワークフロー

## Layer 3: rules

保存するもの:

- `allow`
- `prompt`
- `forbidden`

向いているもの:

- 危険操作の制御
- read/search の無承認化

## Layer 4: automation

保存するもの:

- 定期実行 prompt
- 周期
- 成果物の出力先

向いているもの:

- 週次 review
- 定期的なベストプラクティス棚卸し

## Operating Modes

半自動ループは必ず次の 4 モードに分ける。

1. `Research`
- 外部根拠とローカル現状を読む
- この段階では編集しない

2. `Proposal`
- 候補は最大 3 件
- 各候補に変更内容、利点、リスク、承認要否を付ける

3. `Apply`
- 承認された候補だけ反映する

4. `Review`
- 変更内容、検証内容、残るリスクを返す

この分離を崩すと、調査・提案・反映が混線しやすい。

## Escalate Only If Needed

次の場合だけ `SDK / MCP / App Server` を検討する。

- ボタン式の選択 UI が必須
- 外部データソースの構造化入力が必要
- skill 単体では保持しにくい状態を持ちたい
- 組織横断の中央制御が必要
