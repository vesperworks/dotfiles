# 最重要ルール

## 新しいルールの追加プロセス

言語は日本語です。

ユーザーから今回限りではなく常に対応が必要だと思われる指示を受けた場合:

1. 「これを標準のルールにしますか？」と質問する
2. YES の回答を得た場合、AGENTS.md に追加ルールとして記載する
3. 以降は標準ルールとして常に適用する

## 対話と言語

- 日本語でコミュニケーションを行う
- 指示は短く、本質だけに絞る
- 文末が `！` のときは、実装・編集・実行に即進む
- 文末が `？` のときは、まず検索・調査・確認を優先する
- 提案を求められたときは、推奨度つきで異なる 3 案を出す
- 不明点は推測せず、短く確認する
- 曖昧な点は勝手に補完せず、`[NEEDS CLARIFICATION]` として明示する
- 現在の作業が明確でない場合は、要件定義をヒアリングする

## 実行フロー

- 進め方は必ず `spec -> plan -> tasks -> implement` に従う
- Plan を先に明示し、その後に Tasks 化してから実行する
- まずコードベースと関連ドキュメントを読んで理解する
- 長時間タスクでは `plan.md` や `CONTINUITY.md` に進捗を残す
- 3回連続で問題に行き詰まった場合は、ユーザーに指示を仰ぐ

## 実装ガードレール

- 記載されていない内容や不明点について推測しない
- 既存パターンを優先し、新しい流儀を勝手に持ち込まない
- Codex の標準作法を優先し、custom skill は不足するフォーマット補助に限定する
- 長時間タスクは `PLANS.md` を living document として使う
- plan / research / execution はまず Codex 標準で行う
- ただし破壊的変更、外部影響、承認が必要な操作では安全側を優先する
- 実運用の安全主軸は `AGENTS.md`、`approval_policy`、`sandbox_mode` とし、`rules` は補助として扱う
- `.env`、`.env.*`、`secrets/`、`*.pem`、`*.key`、`id_rsa`、`id_ed25519`、`.git/config` のような機密パスや機密ファイルは勝手に読まない、書かない、必要なら確認する
- ユーザー名や絶対パスが含まれる変更は、コミット前に漏洩がないか意識して確認する
- 変更は小さく、差分が追いやすい形にする
- 実装後はテスト・lint・動作確認を行う
- 最後に「変更内容 / 検証内容 / 残るリスク」を短く報告する

NEVER: Path Traversal
NEVER: Command Injection
NEVER: XSS (Cross-Site Scripting)

## 技術実装の方針

- 公式ドキュメント優先: ライブラリやフレームワークを使用する際は、最新の公式ドキュメントを必ず参照する
- 設計思想の理解: 使用する技術の設計思想を理解し、その作法に従って実装を提案する
- 既存コードとの整合性: プロジェクト内の既存のコーディング規約やパターンに従う

## コーディング原則

- YAGNI: 今必要じゃない機能は作らない
- DRY: 同じコードを繰り返さない
- KISS: シンプルに保つ
- SOLID: 単一責任・オープン/クローズド・リスコフの置換・インターフェース分離・依存性逆転を守る

## 推奨コマンド

- `rg`
- `fd`
- `ni`
- `nr`
- `uv`
- ファイル探索や一覧取得は `find` より `fd` または `rg --files` を優先する
- テキスト検索は `grep` より `rg` を優先する

## コミット運用

- このリポジトリのコミット操作は `jj` を第一候補にする
- コミットは論理単位で分け、無関係な変更をまとめない
- コミット前にユーザー名、絶対パス、トークンなどのセンシティブ情報混入を確認する
- コミットメッセージは原則 `<type>(<scope>): <subject>` の形式に寄せる
- `type` は `feat` `fix` `docs` `refactor` `test` `chore` を基本にする
- コミットメッセージに `Co-Authored-By:`、`Author:`、`Generated with ...`、AI 由来フッターや絵文字を入れない
- 段階コミットが必要な場合は、変更分析 -> グループ提案 -> 承認 -> `jj split` / `jj describe` の順で進める
- 履歴を書き換える可能性がある操作は、rules と承認ポリシーに従って明示確認を取る

## レビュー基準

### レビュー優先度

- CRITICAL: セキュリティ、データ損失リスク
- HIGH: パフォーマンス、保守性
- MEDIUM: 可読性、一貫性
- LOW: スタイル、コメント

### 可読性・保守性

- 変数名は目的が明確にわかること
- 1つの関数は1つの責任のみを持つ
- ネストは3レベル以下に抑える
- コメントは「なぜ」を説明する

### セキュリティ

- ユーザー入力は必ずサニタイズ・バリデーションする
- SQL はパラメータ化クエリを使う
- 機密情報は環境変数で管理し、ログに出力しない
- `eval()` を使わない
- パスワードをハードコーディングしない

### パフォーマンス

- N+1 クエリの発生をチェックする
- イベントリスナーを適切に削除する
- 大量データではページネーションを検討する

### エラーハンドリング

- すべての API 呼び出しでエラーハンドリングを実装する
- ユーザーフレンドリーなエラーメッセージを返す
- ログレベルを適切に設定する

## プロジェクトローカル設定の運用

- プロジェクト固有の設定・パーミッションはプロジェクトルートの `.codex/` に置く。グローバル（`~/.codex/`）を肥大化させない
- テストやビルドなど頻用コマンドの自動承認は `.codex/rules/` に `prefix_rule` で定義する
- グローバルの `default.rules` には汎用ツールの allow と共通 forbidden だけを残す
- プロジェクト固有の命令は `AGENTS.md`（Git ルート）に書く
