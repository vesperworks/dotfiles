# vw-first-principles 第一原理思考フレームワーク

イーロン・マスクが SpaceX / Tesla で用いた「第一原理思考」を、再現可能な 5 フェーズの手順に落としたスキル。

> "Boil things down to the most fundamental truths and then reason up from there, as opposed to reasoning by analogy."
> — Elon Musk, Kevin Rose Foundation interview (2012) / TED (2013)

**核心は 1 つだけ**: 「みんながそうしているから」「ずっとそうだったから」という**類推（analogy）**を前提から排除し、**これ以上疑えない事実**だけを部品にして結論を組み立て直す。

## Core Purpose

ユーザーの問題（コスト・設計・方針・常識への疑問）を、テンプレート表を埋める作業の連鎖として第一原理分析し、「常識の値」と「物理・論理の下限値」のギャップを可視化する。

## 実行原則（必読）

1. **手順がすべて**: 各 Phase のテンプレート表を順に埋める。表が埋まらないうちに次の Phase へ進まない（Exit Criteria を満たすこと）
2. **分類は 3 択**: 前提の出所判定は「事実 / 慣習 / 類推」の 3 択のみ。迷ったら「慣習」に分類して Phase 2 で検証する
3. **数値を取りに行く**: 「素材のスポット価格」「物理的下限」など、検証可能な数値は WebSearch / 計算で実際に取得する。取得できない場合は「未取得」と明記する（推測値をでっち上げない）
4. **実例を見てから始める**: 初回実行時は [worked-example.md](./references/worked-example.md)（バッテリー $600→$80/kWh の完全ワークスルー）を必ず読み、同じ粒度で表を埋める
5. **出典が必要なときは** [sources.md](./references/sources.md) を参照（一次出典の引用集）

## Basic Workflow

### Phase 0: 適用判断（30 秒で終える）

第一原理思考は高コスト。マスク自身も「すべてにこの方法では考えられない。多大な努力を要する」(USC 2014) と認めている。以下の表で判定:

| 質問 | YES なら |
|------|---------|
| 時間制約のある即時判断か？ | **中断** → 類推・既存ベストプラクティスを使う |
| 長年最適化され尽くした確立領域のルーティンか？ | **中断** → 既存手法に従う |
| 「何か新しいこと」をやろうとしているか？ コスト・構造を根本から変えたいか？ | **続行** → Phase 1 へ |
| 「相場」「常識」「みんなそうしてる」が根拠の結論に違和感があるか？ | **続行** → Phase 1 へ |

中断する場合もユーザーにその旨を 1 行で伝える（「この問題は類推で解くのが正しい」も第一原理的な判断である）。

**Exit Criteria**: 続行/中断の判定理由を 1 行で言語化した。

### Phase 1: 前提マッピング（Assumption Mapping）

問題に含まれる前提・思い込みを**すべて**書き出し、出所を 3 択分類する:

| # | 前提（〜だから〜だ） | 出所 | 検証方法 |
|---|---------------------|------|----------|
| 1 | バッテリーパックは $600/kWh が相場だから EV は高い | 類推 | 素材コストを積算する |
| 2 | ... | 事実 / 慣習 / 類推 | ... |

- **事実**: 物理法則・数学・確定した測定値（→ Phase 2 でそのまま部品になる）
- **慣習**: 「ずっとそうだったから」（→ 要検証）
- **類推**: 「他社・他者がそうしているから」（→ 要検証）

掘り下げには「5 つのなぜ」を使う: 前提に対して「なぜ？」を最大 5 回繰り返し、答えが「事実」か「それ以上理由がない（＝慣習/類推）」に到達するまで掘る。

**Exit Criteria**: 前提が 3 件以上列挙され、すべてに出所ラベルが付いた。

### Phase 2: 根本真実への分解（Decomposition）

「慣習」「類推」ラベルの前提を剥がし、**これ以上疑えない事実**だけを残す。事実は 3 カテゴリで整理:

| カテゴリ | 例 | この問題では |
|----------|-----|------------|
| 物理的制約 | 熱力学、材料強度、光速 | ... |
| 数値的事実 | 素材のスポット価格、API レート上限、人月単価 | ... |
| 論理的必然 | 「N 件処理には最低 N 回の読み取りが要る」 | ... |

**コスト問題なら Idiot Index を計算する**:

```
Idiot Index = 完成品（現状）のコスト ÷ 原材料（根本要素）のコスト合計
```

- 比率 1〜2 倍: 既に最適。第一原理での攻め所は薄い
- 比率 5 倍以上: プロセス・流通・慣習にムダがある。再構築の価値が高い
- SpaceX のロケットは 50〜100 倍だった（→ 内製化の根拠になった）

**Exit Criteria**: 「事実」だけのリストが完成し、各項目に根拠（出典 URL / 計算式 / 物理法則名）が付いた。数値が取れなかった項目は「未取得」と明記した。

### Phase 3: ゼロからの再構築（Reconstruction）

Phase 2 の事実**だけ**をパズルのピースとして、白紙から解を組み立てる。

手順:
1. 目的を 1 文で再確認する（「何を達成したいのか」— 手段ではなく目的）
2. 「この事実・制約だけを与えられた新規参入者なら、どう作るか？」を 3 案以上発散する（既存解の改善案は禁止。ゼロからの構成案のみ）
3. 各案を Phase 2 の事実と突き合わせ、矛盾する案を消す

| 案 | 構成 | 理論コスト/性能 | Phase 2 の事実との矛盾 |
|----|------|----------------|----------------------|
| A | ... | ... | なし / あり（→棄却） |

**Exit Criteria**: 既存のやり方をコピーしていない案が 2 つ以上残り、それぞれに理論値（コスト下限・性能上限）が付いた。

### Phase 4: 検証と反証（Hypothesis Testing）

導出した結論を壊しに行く。以下の 3 問に答える:

1. **反証条件**: 「この結論が間違っているとしたら、どんな事実が存在するはずか？」→ その事実を実際に探す（WebSearch / 実測 / 計算）
2. **未知の未知**: 「自分が知らないことで、この前提を崩す情報はないか？」（規制・補助金・特許・隠れた運用コスト等、Phase 2 のリストに**載っていない**カテゴリを疑う）
3. **抽象レベル**: 「論理的に正しいが、実用の抽象度に合っているか？」（理論下限 $80/kWh は「いつか到達可能」であって「明日実現」ではない）

反証に成功した場合は Phase 1〜2 に戻り、見落とした前提を表に追加して再実行する（最大 2 周。3 周目に入る場合はユーザーに状況を報告して判断を仰ぐ）。

**Exit Criteria**: 3 問すべてに回答し、生き残った結論に「確度（高/中/低）」と「次の検証アクション」が付いた。

## Output Deliverables

最終レポートを以下の構成でユーザーに提示する（長大な分析になった場合は `.brain/thoughts/shared/research/{date}-first-principles-{topic}.md` への保存を提案）:

```markdown
# 第一原理分析: {テーマ}

## TL;DR
- 常識の値: {X} / 第一原理の下限値: {Y} / ギャップ: {N 倍}（Idiot Index）
- 結論: {1-2 文}（確度: 高/中/低）

## Phase 1: 剥がした前提（事実 n 件 / 慣習 n 件 / 類推 n 件）
{表}

## Phase 2: 残った根本真実
{表 + 出典}

## Phase 3: ゼロベース再構築案
{表}

## Phase 4: 反証テスト結果
{3 問への回答}

## 次のアクション
{検証アクション 1-3 件}
```

## Rollback / Recovery（分析が空転した場合）

- **前提が列挙できない**: 問題が抽象的すぎる。ユーザーに「疑いたい常識・相場・コスト」を 1 つ具体的に挙げてもらう
- **数値が取れない**: 「未取得」と明記して論理構造だけ提示する。でっち上げは [failure-modes.md](./references/failure-modes.md) の「不完全な原理からの推論」を引き起こす
- **3 周しても反証が止まらない**: 未知の未知が多すぎる領域。第一原理思考の適用限界としてユーザーに報告し、類推アプローチへの切替を提案する
- **スキル破損時**: `git restore claude/.claude/skills/vw-first-principles/`（dotfiles リポジトリ内）
