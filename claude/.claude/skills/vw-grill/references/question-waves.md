# Question Waves 設計

## Overview

wave 方式（基礎→明確化→深掘り）の質問設計詳細。各 wave は AskUserQuestion で1回のバッチとして提示する。

## AskUserQuestion の使い方

- 1 wave = AskUserQuestion 1回（最大4問）
- 各質問の**先頭オプションは推奨案**とし、label に「(推奨)」を付ける。推奨案は Phase 0 の調査結果から導き、description に根拠を1行書く
- 選択肢で表現できない自由回答が必要な質問は、オプションを「代表案2-3個」にして Other に委ねる
- 色絵文字は使わない。識別子は数字（グローバルルール準拠）

## Wave 0: ゼロ次質問（対象が特定できないときのみ、1問）

お題に「何についての話か」が無い場合、G/S/O テンプレートは適用できない（すべて対象の存在を前提にしているため）。
先に1問だけ出す: 「何についての話ですか？」— 選択肢は直近の作業文脈（jj log・TaskList・アクティブ PRP）から推奨案を作る。
対象が確定したら Phase 0 の調査からやり直す。

## Wave 1: 基礎（3-4問）

未充足の G / S / O を優先する。**問数の決め方**: 未充足の G/S/O で最大3問。4問目は Phase 0 調査で見つかった
最大の assumption（推定のまま進めると危ないもの）1問に使う。該当がなければ3問で止める。

質問テンプレート:

| 観点 | 質問の型 | 推奨案の導き方 |
|------|---------|---------------|
| G | 「この作業が成功したと言える状態は、次のどれに最も近いですか？」 | 会話中の動詞（自動化したい/直したい/知りたい）から成功状態を具体化 |
| S | 「今回やらないこと（範囲外）はどれですか？」（multiSelect 可） | 関連するが工数が膨らむ隣接タスクを列挙して除外候補にする |
| O | 「成果物はどの形が使いやすいですか？」 | 過去の類似成果物（.brain 内）と同じ形式を推奨 |

## Wave 間の要約（必須）

各 wave の回答後、次を3行で提示してから次 wave に進む:

```
facts: {今 wave で確定したこと}
assumptions: {推定のまま残っていること（次 wave の質問候補）}
risks: {回答から新たに見えた懸念}
```

assumptions と risks が次 wave の質問の材料になる（回答を踏まえて質問を組み立てる）。

## Wave 2: 明確化（2-3問）

C / V と、Wave 1 の回答間の矛盾・依存を突く:

| 観点 | 質問の型 |
|------|---------|
| C | 「制約として効くのはどれですか？」（環境/互換性/時間/コスト/既存資産、multiSelect） |
| V | 「完成の判定は何で行いますか？」（テスト実行/実機確認/独立レビュー/ユーザー確認） |
| 矛盾 | 「Wave 1 で X と Y を選びましたが、両立には Z の判断が必要です。どちらを優先しますか？」 |

## Wave 3: 深掘り（1-2問、未充足観点のみ）

暗黙の前提と盲点を突く。devil's advocate レンズ:

- 「この前提（{根拠なしの assumption}）が崩れた場合の代替はありますか？」
- 「R が空です。過去の類似作業（{.brain の実例}）では {実際の落とし穴} がありました。今回は該当しますか？」

## ゲート再判定の手順（wave 後に毎回）

1. 6観点の表を再生成し、各行に充足/未充足 + 根拠1行を書く
2. 根拠が「〜のはず」「一般的に〜」の形なら未充足に戻す（推測は根拠ではない）
3. 全充足 → Phase 3 へ。未充足あり → 次 wave（上限 5 wave）

## Common Pitfalls

- **読まずに質問する** → 質問が浅くなり、ユーザーが既に言ったことを聞いて信頼を失う。Phase 0 を飛ばさない
- **一度に全部聞く** → 回答疲れで品質が落ちる。wave 分割と推奨回答で1回の負荷を下げる
- **推測で観点を埋める** → 「賢い補完」は spec の誤りとして後工程に伝播する。不明は質問するか未決に落とす
- **無限インタビュー** → 5 wave 上限で強制終了し、未決事項として明示する
