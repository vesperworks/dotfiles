# vw-grill — 質問攻めによる仕様導出

## Core Purpose

曖昧な「やりたいこと」を、最小回数の質問で検証可能な仕様（spec.md）に変換する。
モデルの推論力に依存しない手順駆動設計: 観点チェックリスト + ゲート式判定 + 質問テンプレートで回す。

## 設計原則（3つ）

1. **読まずに質問しない**: コード・ファイル・memory で分かることはユーザーに聞かない。先に読むほど質問が鋭くなり、質問数が減る
2. **推奨回答を必ず付ける**: 各質問に文脈から導いた推奨案を先頭に置く。ユーザーは yes と言うだけで進められる（回答コスト最小化）
3. **ゲート式判定**: 観点の充足は「根拠（ユーザー発言 or 実ファイルの引用）が1行添えられている」場合のみ。加算式・部分一致・「賢い補完」は禁止。不明は不明のまま未充足に置く

## Phase 0: 先に読む（質問前の必須調査）

お題を受け取ったら、質問する前に以下を調査する（read-only）。
`{project}` は **git/jj リポジトリルートのディレクトリ名で固定**する（cwd がネストしていてもルートまで遡る）。

0. **ゼロ次判定**: お題から「何についての話か（対象）」が特定できない場合（例:「なんかいい感じにしたい」）は、
   以下の調査をスキップし、Wave 1 の前に**ゼロ次質問1問**（「何についての話ですか？」+ 直近の作業文脈から推奨案）だけを先に出す。対象が決まってから Phase 0 をやり直す
1. 会話履歴 — ユーザーが既に言ったことを再質問しない
2. 対象コード・設定ファイル — 探索の起点は現在のリポジトリルート。お題の名詞・動詞で `grep -ril` / `ls` し、ゼロヒットなら「該当なし」と記録する（形式的に全源をチェックし、結果を残す）
3. **CLAUDE.md（グローバル + プロジェクト）と `.claude/rules/`** — C（制約）と R（リスク）の根拠になる既存ルール（例: bash 3.2 互換、trash 使用）はここにある
4. `MEMORY.md` と関連 memory — ユーザーの既知の preferences・過去の決定
5. `.brain/{project}/` の PRP・spec・HANDOFF — 過去の関連決定
6. `jj log` / `git log` — 直近の作業文脈

調査で埋まった観点には根拠として **出典（file:line or 発言の要約）を記録**する。

## Phase 1: 観点ゲートの初期判定

6観点それぞれを、Phase 0 の調査結果で充足/未充足に判定する:

| # | 観点 | 充足の条件（ゲート） |
|---|------|---------------------|
| G | Goal | 目的と成功基準が1行で書け、根拠がある |
| S | Scope | やること/やらないことの境界が書け、根拠がある |
| C | Constraints | 環境・技術・時間・コストの制約が列挙され、根拠がある |
| O | Output | 成果物の形式・保存先が特定でき、根拠がある |
| V | Verification | 「何を実行すれば完成と判定できるか」がコマンド or 手順1行で書ける |
| R | Risks | 落とし穴・エッジケース・依存が最低1件挙がっている（「なし」も根拠付きなら可） |

判定表を作り、**未充足の観点だけ**を Phase 2 の質問対象にする。
全観点が既に充足なら質問をスキップして Phase 3（spec 生成）へ直行してよい（その旨をユーザーに1行報告する）。

## Phase 2: Wave 方式の質問

AskUserQuestion で 1 wave ずつバッチ提示する。詳細な質問設計は
[question-waves.md](./references/question-waves.md) を必ず読むこと。

- **Wave 1（基礎）**: 3-4問。未充足の G/S/O を優先。各質問の先頭オプションは推奨案 +「(推奨)」表記
- **Wave 間の要約**: 各 wave 後に facts（確定）/ assumptions（推定、要確認）/ risks（新たに見えた懸念）を3行で提示
- **Wave 2（明確化）**: 2-3問。C/V と、Wave 1 の回答から生じた矛盾・依存
- **ゲート再判定**: Wave 2 後に6観点を再判定。全充足なら Wave 3 をスキップ
- **Wave 3（深掘り）**: 1-2問。未充足観点のみ。暗黙の前提・矛盾を突く
- **上限**: 最大 5 wave。到達したら未充足観点を「未決事項」として spec に落とし、強制終了する（無限インタビュー防止）

## Phase 3: spec.md 生成

全観点充足（or wave 上限到達）で、[spec-template.md](./references/spec-template.md) に従い生成する:

- 保存先: `.brain/{project}/spec/{YYYY-MM-DD}-{slug}.md`
- 決定事項には**根拠列**（ユーザー回答 / file:line / 推奨案承認）を必ず付ける
- 未決事項は「未決 + 判断に必要な材料1行」形式で分離する（勝手に埋めない）
- 書き込み後 `ls -la <path> && wc -l <path>` を実行し、出力を報告に含める

## Phase 4: 後続ルーティング

spec の性質から次の一手を推奨確率付きで提案する:

| specの性質 | ルーティング先 |
|-----------|---------------|
| 反復・自動化・監視系 | `/vw-loop-architect`（spec.md を入力として渡す） |
| 機能実装系 | `/vw:plan-prp`（spec.md を要件として渡す） |
| セッション引き継ぎ系 | `/vw-handoff` |
| 単発・小規模 | 直接実行を提案 |

ルーティングは提案のみ。実行はユーザーの合意後（Harper Reed 方式: フェーズ遷移は明示的な合図で）。

## Rollback / Recovery

- 質問が的外れと言われたら: Phase 0 の調査不足が原因。該当領域を読み直してから wave をやり直す
- spec が間違っていたら: spec.md は上書きせず `-rev2` を付けて再生成（旧版は判断履歴として残す）
- ユーザーが疲れたら: その時点の確定分で spec を出し、残りを未決事項に落とす
