# WIKI SCHEMA — {topic}

この wiki の唯一のルールブック。エージェント（Claude Code / Cursor / Codex 等、何で開いても同じ）は、
この wiki に書き込む前に必ずこのファイル全体を読み、以下の規則に従うこと。

## 構造

```
raw/             # git の不変性が使えないソースの置き場（R1b）。追加後は不変。エージェントは既存ファイルに書き込まない
wiki/
├── SCHEMA.md    # このファイル（ルールブック）
├── SOURCES.md   # 監視ソース登録簿（git 管理下の原本パスを列挙。sync の対象）
├── INDEX.md     # 全ページカタログ（カテゴリ別 + 1行要約）
├── LOG.md       # 追記専用の操作ログ
├── inbox/       # 分類に迷ったページの一時置き場
├── concepts/    # 概念・手法・パターン（オントロジー層の本体）
├── entities/    # 人物・組織・ツール・モデル・プロダクト
├── sources/     # ingest した原本 1 本につき 1 要約ページ
└── queries/     # 価値ある Q&A のアーカイブ
```

## 10 か条

1. **R1 ソースは 2 種・どちらも不変**: (a) **git 参照ソース** = リポジトリ管理下の原本。raw/ にコピーせず原本を読み、取り込んだ版のコミット SHA とコミットログ 1 行を記録する（下記「ソース記録形式」）。不変性は git の履歴が保証する。 (b) **raw/ ソース** = git の不変性が使えないもの（PDF・URL 退避・git 管理外ファイル・このトピックで新規生成される記録の正本）。raw/ への新規追加は人間または合意済みワークフローが行い、追加後のファイルは不変。エージェントは raw/ の既存ファイルに書き込まない。`wiki/` はエージェントが全権管理し、人間は原則直接編集しない（SOURCES.md のみ人間が編集してよい）。
2. **R2 操作は 4 つだけ**: ingest / query / lint / sync。手順は下記の通り厳守。
3. **R3 1 エンティティ = 1 ファイル**: ファイル名は kebab-case。名前衝突時は修飾子を付ける（例: `attention-cnn.md`）。`sources/` のみ `YYYY-MM-DD-<slug>.md` 形式（原本ファイル名に日付があればそれ、なければ初回 ingest 日。概念ページとの slug 衝突防止）。**原本 1 本につき sources ページは 1 つ** — 同一原本の新しい版は新ページを作らず既存ページを更新する。**ページを作る前に必ず INDEX.md を検索し、既存ページの更新を新規作成より優先する。**
4. **R4 二層モデル**: 各ページは上下 2 層。上層 = Compiled Truth（現時点の最善理解。追記禁止。証拠が変わったら**全面書き直し**。矛盾せず情報が増えただけでも、上層に反映する価値があれば書き直してよい）。下層 = Timeline（`- **YYYY-MM-DD** | 出来事/出典` の**追記のみ**、既存行の編集禁止）。Compiled Truth の主張はすべて Timeline の行に根拠を持つこと。
5. **R5 wikilink はフルパスのみ**: `[[concepts/attention]]` 形式。短形式 `[[attention]]` は禁止。
6. **R6 frontmatter は 6 フィールドのみ**: `type / aliases / related / sources / updated / confidence`。フィールドを増やさない。
7. **R7 ページ化の抑制と昇格**: 新しい概念・エンティティに出会っても、**即ページ化してよいのは「そのソースの主題そのもの」（タイトルに現れるレベル。複数あれば各々）だけ**。それ以外の言及は INDEX.md の「## 昇格候補」に `- <slug>: 登場箇所1, 登場箇所2` 形式でカウントを記録する。**異なる 3 ページに登場した時点で**ページに昇格させ、候補行を消し、登場箇所すべてからリンクを張る。1〜2 回しか登場していない語にページを作らない（ファイル爆発防止）。
8. **R8 配置は 1 か所**: 置き場所は下記デシジョンツリーで 1 つに決める。10 秒迷ったら `inbox/YYYY-MM-DD-<slug>.md` に置き、次の lint で振り分ける。
9. **R9 出所明示**: ソースに書かれていない主張（推論・モデルの一般知識を含む）には文末に `(inferred)` を付ける。**異なるソース同士**が矛盾する情報を持つ場合は勝手に上書きせず、両論併記して `confidence: low` にする。ただし**同一原本の新しい版**が古い版と食い違う場合は矛盾ではなく更新 — 新しい版を優先して Compiled Truth を書き直し、経緯は Timeline に残す。
10. **R10 同期と日付**: ページの作成・改名・削除をしたら、同じ作業の中で INDEX.md を更新し、LOG.md に 1 行追記する。後回しにしない。**Timeline 行・LOG・frontmatter `updated` の日付はすべて操作実行日（今日）**。ソース自体の日付は本文と `sources:` で表現する。

## ソース記録形式（git 参照ソース）

frontmatter `sources:` と Timeline 行で共通の形式: `<リポジトリ相対パス> @ <short-SHA> "<コミットログ 1 行>"`

- 取得コマンド: `git log -1 --format='%h %s' -- <path>`
- 例: `KYC/_context/people/yamada.md @ a1b2c3d "docs(kyc): 山田の異動を反映"`
- raw/ ソースは従来どおりファイル名のみ、外部ソースは URL でよい。

## 配置デシジョンツリー

```
それは「考え方・手法・パターン」か？        → concepts/
それは「固有名詞（人・組織・ツール・モデル）」か？ → entities/
それは「読んだソース 1 本の要約」か？        → sources/
それは「質問への回答」か？                  → queries/
どれでもない / 迷う                         → inbox/
```

## ページテンプレート（全ページ共通）

```markdown
---
type: concept | entity | source | query
aliases: []
related: []          # 拡張子なしフルパスの配列（例: [concepts/rag, entities/claude]）
sources: []          # git 参照ソースは「ソース記録形式」、raw/ 内ファイルはファイル名、外部は URL
updated: YYYY-MM-DD  # 操作実行日（ソースの日付ではない）
confidence: high | medium | low
---
```

## 操作手順

### ingest（ソース 1 本を wiki に反映する）

順番に実行し、各番号を飛ばさない:

1. 対象ソースの所在を確認する: リポジトリ管理下の原本（git 参照ソース）か、`raw/` 内のファイルか。どちらでもなければ人間に配置を依頼して停止。
2. git 参照ソースの場合: `git status --porcelain -- <path>` で未コミット変更を確認し、あれば人間にコミットを依頼して停止（取り込むのはコミット済みの内容だけ）。次に `git log -1 --format='%h %s' -- <path>` で SHA とコミットログ 1 行を取得する。
3. ソースを読み、sources ページを用意する: 同じ原本の sources ページが既にあれば（= 新しい版の取り込み）Timeline に追記し、Compiled Truth を書き直す（R4, R9）。なければ `sources/YYYY-MM-DD-<slug>.md` をテンプレートで新規作成する（type: source）。出所は「ソース記録形式」で `sources:` と Timeline に書く。
4. ソースに登場したエンティティと概念を箇条書きで列挙する（5〜15 個目安）。
5. 列挙した各項目について INDEX.md（「## 昇格候補」セクション含む）を検索し、3 分岐する:
   - 既存ページあり → Timeline に追記（日付は今日、R10）。上層に反映すべき新情報なら Compiled Truth を書き直す（R4）。
   - 既存ページなし & **このソースの主題そのもの** → デシジョンツリーで配置を決めて新規作成（R3, R7, R8）。
   - 既存ページなし & 単なる言及 → INDEX.md の「## 昇格候補」にカウントを追記。**ページは作らない**（R7）。
6. R7 昇格チェック: 昇格候補のうち登場箇所が 3 ページに達したものを昇格させ、候補行を消す。
7. 関連ページ同士に相互の `related:` と本文 wikilink を張る（R5）。**リンクは実在するページにのみ張る**。未作成の昇格候補にはリンクせず、INDEX.md の候補行に任せる。新規ページの本文から `sources/` の要約ページにもリンクを張る。
8. INDEX.md に新規・変更ページを反映し、LOG.md に `## YYYY-MM-DD ingest | <source-slug>` を追記する（日付は今日、R10）。

### sync（監視ソースの更新を検知して一括 ingest）

1. `wiki/SOURCES.md` の監視パス一覧と、LOG.md の直近の sync 行に記録された前回 HEAD SHA を読む。sync 記録がなければ人間に基準（特定の SHA か、登録パスの全件 ingest か）を確認して停止。
2. `git diff --name-only <前回SHA>..HEAD -- <監視パス...>` で、前回以降に追加・更新されたファイルを列挙する。`wiki/` 配下のファイルが混ざっていたら除外する。
3. `raw/` 内で対応する sources ページがないファイルがあれば列挙に加える。
4. 列挙が 5 件を超える場合は、先に LOG.md へ取り込み予定の一覧を追記してから着手する（途中失敗からの復旧用）。
5. 各ファイルを上記 ingest 手順で順に取り込む。
6. LOG.md に `## YYYY-MM-DD sync | HEAD <short-SHA>（N 件）` を追記する（R10）。

### query（質問に wiki で答える）

1. INDEX.md を読み、関連カテゴリのページを開いて回答を合成する。ソース原本（git 参照・raw/）は wiki に答えがない場合のみ参照する。
2. wiki に答えがなかった部分は「wiki 未収載」と明示する（捏造しない）。
3. 回答が再利用価値を持つ場合（比較・調査・意思決定）、`queries/<slug>.md` に file-back し、INDEX.md と LOG.md を更新する（R10）。

### lint（健全性検査。「lint して」と言われたら実行）

以下の 6 点を検査し、結果を表で報告する。**修正の実行は人間の確認後**（提案までが lint の仕事）:

1. INDEX.md と実ファイルの差分（INDEX 未掲載ページ / 実体のない INDEX 行）
2. リンク切れ wikilink（フルパスが実ファイルに解決しないもの。`related:` の値も対象）
3. 孤立ページ（他ページの本文 wikilink にも `related:` にも現れないページ。INDEX.md への掲載はリンクとみなさない。`sources/` と `queries/` は対象外）
4. 重複の疑い（同一概念とみられる複数ページ）→ マージ案を提示
5. `inbox/` の滞留ページ → 振り分け案を提示。`confidence: low` のページ一覧 → レビューキューとして提示
6. SOURCES.md の監視パスが実在するか / sources ページの出所が「ソース記録形式」に従っているか
