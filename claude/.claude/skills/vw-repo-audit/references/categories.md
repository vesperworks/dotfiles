# 監査カテゴリ・ルーブリック（6 カテゴリ）

各 Sonnet エージェントは担当カテゴリの節だけを読み、その観点**のみ**で監査する。

## 全カテゴリ共通の規約（必読）

- **証拠必須**: すべての指摘に file:line（行が特定できないものは file とその根拠）を付ける。
  読んでいないファイルについて推測で指摘しない
- **1 根本原因 = 1 指摘**: 同じ原因から派生する症状はまとめ、影響箇所を file:line リストで列挙
- **反証してから昇格**: コマンド実行で白黒つけられる仮説は実行して確認する。実行できない場合は
  「未検証の仮説」と明記し検証コマンド 1 行を添える
- **機密は絶対にマスク**: API キー・トークン・パスワードらしき値を発見しても、**値そのものを
  所見・構造化出力・ファイルのどこにも書かない**。file:line + 種別（例: "AWS キーらしき文字列"）のみ
- **「問題なし」も成果**: 指摘ゼロの場合は「何をどう確認して問題なしと判断したか」を根拠として書く
- **審査対象**: 渡された差分ファイル・表層ファイル・コミットログの範囲のみ。repo 全探索はしない

## §1 First Impression（第一印象）— 配点 20

初見の訪問者が 30 秒で「これは何か・使えるか」を判断できるか。

- README: 冒頭 3 行で「何をするものか」が分かるか / インストール・使い方の手順があるか /
  スクリーンショットや例があるか（CLI/設定系なら例示コード）
- LICENSE ファイルの存在と内容の整合（README の記載と一致するか）
- リポジトリ直下の第一印象: ルートに雑多なファイルが散乱していないか
- バッジ・リンク切れ（明らかに壊れた相対リンク）
- Critical 例: LICENSE 欠落（公開 OSS として法的に不備）/ README が存在しない
- Serious 例: README が実態と乖離した説明で始まる / インストール手順欠落
- Nitpick 例: スクリーンショットが無い / バッジが古い

## §2 Hygiene（衛生）— 配点 30

公開してはいけないものが混ざっていないか。**最重要カテゴリ**。

- 機密: API キー・トークン・パスワード・秘密鍵らしき文字列（値はマスクして報告）
- 個人情報: ユーザー名のハードコード（ホームディレクトリ直書きの絶対パス等）、メールアドレス、
  社内情報・取引先名・内部 URL
- .gitignore の網羅性: セッション副産物（ログ、キャッシュ、`.DS_Store`、エディタ一時ファイル）が
  追跡対象に入り込む穴がないか
- ゴミファイル: `*.log`、`*.bak`、`*~`、実験の残骸、コメントアウトされた大きな死骸コード
- Critical 例: 機密値・秘密鍵の混入 / 社外秘情報
- Serious 例: ユーザー名入り絶対パスのコミット / ログファイルの追跡
- Nitpick 例: .DS_Store が gitignore に無い（未混入）

## §3 History（履歴）— 配点 15

公開される履歴（コミットログ）がプロとして読めるか。

- コミットメッセージの一貫性: conventional commits 率、言語の統一
- 恥ずかしいメッセージ: "wip"、"fix2"、"あとで直す"、意味のない "update"
- 巨大コミット（無関係な変更の抱き合わせ）、逆に無意味な 1 行連投
- コミットメッセージ内の機密・個人情報・感情的な表現
- divergent・無名の浮遊コミット・未整理の実験ブランチ（jj なら `jj log` のヘッド、
  git なら未マージブランチ）が push 対象に混ざっていないか
- Critical 例: メッセージに機密や攻撃的表現
- Serious 例: "wip"/"fix2" が push 対象に複数 / 無名コミットが混入
- Nitpick 例: type(scope) 形式の揺れが少数

## §4 Structure（構造）— 配点 15

ディレクトリ・ファイル配置が自己説明的か。

- 命名の一貫性: kebab-case/snake_case の混在、複数形/単数形の揺れ
- README に説明のないトップレベルディレクトリ
- 死んだファイル: どこからも参照されない設定・スクリプト（判定できたものだけ。
  参照検索を実行し、その結果を根拠に付ける）
- 重複: ほぼ同一内容のファイルが複数箇所にある
- Critical 例: （構造単独で Critical は稀 — 機密系は §2 に委譲）
- Serious 例: 説明なき謎ディレクトリが複数 / 明らかな死骸ファイル
- Nitpick 例: 命名スタイルの軽微な揺れ

## §5 Docs Freshness（ドキュメント鮮度）— 配点 15

書いてあることと実態が一致しているか。**検証可能な乖離のみ**指摘する。

- README・ドキュメントに書かれたコマンド/パス/ファイルが実在するか（実際に ls/grep で確認）
- 「進捗表」「パッケージ一覧」等の列挙と実ディレクトリの突合
- ドキュメントに書かれた手順の前提（依存ツール等）が現状と一致するか
- Critical 例: インストール手順が現状で全く動かない（検証済みの場合のみ）
- Serious 例: 存在しないファイル/コマンドへの言及が複数
- Nitpick 例: 軽微な記述の古さ（機能に影響なし）

## §6 Taste（センス）— 配点 5

「ダサくないか」。**自前の審美眼を使わず、著名 OSS の慣習との差**で測る
（例: dotfiles 分野なら mathiasbynens/dotfiles、holman/dotfiles 等の慣習）。

- 絵文字の過剰/不統一（README 見出し全部に絵文字 vs ゼロ、の混在）
- 言語の混在に一貫性があるか（日本語 README なら統一されているか。日本語であること自体は減点しない）
- 誇大な形容詞（"blazingly fast" 的な根拠なき自称）
- コメント・ドキュメントのトーンの揺れ
- **全て Nitpick 扱いが基本**。Serious は「著名 OSS の慣習と明確に反し初見の信頼を損なう」場合のみ
