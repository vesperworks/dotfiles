# HANDOFF Rubric — 5基準×5点（25点満点）

## Overview

vw-handoff 成果物の採点基準。model-bench（2026-07-07）の rubric 形式を踏襲:
各基準に「5点の条件」と「0点の条件」をペアで定義し、採点手順（実測照合）を同梱する。
**合格基準: 全基準 4点以上**（合計点でなく最低点で判定する — 1基準の崩壊は引き継ぎ事故に直結するため）。

## 基準

| # | 基準 | 5点の条件 | 0点の条件 |
|---|------|----------|----------|
| 1 | 網羅性 | 素材（会話タスク・TaskList・jj log）の全項目が本文か「省いたもの」のいずれかに現れる（サンプリング照合で欠落ゼロ） | 大量の項目が黙って消失している |
| 2 | 裏付け | 「完了」全件に実在する hash、状態記述にツール実出力が随伴。「未確認」「不明」の分類が正しく使われている | hash なしの「完了」や、出典のない状態記述が複数ある |
| 3 | 実行可能性 | 未完了タスク全件に実行可能な検証コマンド/手順1行があり、TL;DR の次の一手が初見でも実行できる | 検証方法なし、次の一手が抽象的（「続きをやる」等） |
| 4 | 鮮度管理 | expires_when・時点明記・適用状態宣言がすべて揃い、stale 化の検知が可能 | 日付なし・破棄条件なし（腐って有害情報化するタイプ） |
| 5 | 索引性 | TL;DR が先頭にあり、初見の AI が数分で現在地と次の一手を判断できる。詳細は外部パス参照で本文が軽量 | 時系列の作業ログ羅列で、どこから読めばいいか分からない |

## 採点手順（verifier はこの順に実測すること）

1. **基準2**: 「完了タスク」表から hash を3件（3件未満なら全件）抽出し、`git log --oneline | grep {hash}`（または `jj log` 相当）で実在確認する。実在しない hash が1件でもあれば基準2は 0点
   - 例外: jj の working copy commit id は `git log` に現れない。文書がこれを hash として使っていたら（「未コミット完了」表以外で）基準2の減点対象とするが、「実在しない hash」（0点条件）とは区別し、`git cat-file -t {id}` で object 自体の実在は確認すること
   - 「未コミット完了」表の裏付け（ls/wc/jj status の実出力）は、記載のコマンドを再実行して照合する
2. **基準1**: 採点者に渡された素材リスト（または `git log --oneline -20` と会話要約）から10件（不足時は全件）を抽出し、HANDOFF 本文 or「省いたもの」に現れるか照合する。欠落率で採点（0件=5点、1-2件=3点、3件以上=0-1点）
3. **基準3**: 未完了タスク表の検証コマンド上位3件を読み、(a) 構文的に実行可能か (b) そのタスクの完了を実際に判定できる内容か、を確認する。可能なら1件を dry-run（read-only のもののみ）
4. **基準4**: frontmatter に date / expires_when / audience、本文に「時点」表記と適用状態宣言があるか機械的に確認
5. **基準5**: 文書の先頭 30 行だけを読み、「今何をすべきか」を1行で言えるか自問する。言えなければ 3点以下
6. 各基準のスコア + 1行の根拠 + 修正指示（3点以下の基準のみ、具体的な行とやり直し方）を出力する

## 採点者の制約

- 採点者は fresh-context のサブエージェント（Sonnet 以上。Haiku 不可）
- maker の推論過程・自己評価は渡さない（成果物 + 素材 + この rubric のみ）
- 「よく書けている」等の総評は不要。基準ごとのスコアと根拠のみ
