# vw-handoff — 検証可能なセッション引き継ぎ書の生成

## Core Purpose

「次のセッションを開いた初見の AI が、数分で状況を把握し、最初の一手を間違えずに打てる」文書を生成する。
実測ベンチ（model-bench 2026-07-07）の結論に基づき、**明示的な手順 + rubric + 独立 verifier** でモデル差を吸収する設計。自己申告・推測・精神論を許さない。

## 絶対ルール（生成中つねに適用）

1. **「解決済み/完了」はコミット hash 併記時のみ**。`jj log`（または `git log --oneline -- <file>`）で特定できなければ「未確認」に分類する。曖昧なら Open 側に倒す
2. **状態記述はツール出力の転記のみ**。「〜のはず」「おそらく」を書きそうになったら、実測するか「未検証の仮説 + 検証コマンド1行」として分離する
3. **精神論の禁止**: 「注意する」「気をつける」「意識する」を含む行は書かない。すべて「発動条件 → 手順 → 根拠」の3点形式に変換する
4. **不明は不明と書く**。賢い補完・もっともらしい推測は引き継ぎ事故の主因。埋められないセルは「不明（{何が分かれば埋まるか}）」と書く。**例外: TL;DR の「次の一手」と未完了タスク上位3件の検証方法には不明を許さない** — read-only コマンドで特定を試み（ls / grep / git log）、それでも埋まらなければ「特定作業そのもの」を最初の一手として書く（例: 「検証: まず `ls {候補パス}` で成果物の所在を特定する」）
5. **working copy は hash ではない**: jj の working copy commit id（`jj status` の @ 行）は `git log` に現れず、後続セッションが検証できない。未コミットの完了はテンプレートの「未コミット完了」表に、ツール実出力を裏付けとして書く

## Phase 0: 素材収集（read-only）

以下を順に収集し、素材リストを作る:

1. 会話履歴のタスク・決定・エラーと修正（ユーザー指示の原文を優先）
2. `TaskList`（あれば）— 未完了/完了の一次ソース
3. `jj log`（または `git log --oneline -20`）と `jj status` — 実出力を保存しておく（後で転記する）
4. 今セッションで touch したファイル一覧
5. 関連 memory・`.brain/{project}/` の関連文書（正の所在を特定する)

**再実測の義務**: 素材・memory・会話に古い数値/状態（例: 「origin より N コミット ahead」「〜は未実施」）が
あり、read-only コマンド1本で再実測できるものは、**転記前に再実測して現在値に更新する**
（乖離があれば「記録時 N → 現在 M」と両方書く）。セッション途中の記録は書いた瞬間から古くなる。

## Phase 1: 充足度ゲート → 不足だけ質問

8観点を判定する。充足の条件は「素材に根拠がある」こと（ゲート式。推測での充足は禁止）:

| # | 観点 | 充足の条件 |
|---|------|-----------|
| 1 | 現在の作業と意図 | ユーザーの元の依頼と現在地が1行で書ける |
| 2 | 未完了タスク + 優先度 | 各件に「次に何をするか」がある |
| 3 | 完了タスク + hash | 各件に hash があるか「未確認」の判定ができる |
| 4 | エラーと修正 | 今セッションで踏んだエラーと解決策が列挙できる |
| 5 | 落とし穴 | 次セッションが踏みそうな地雷が根拠付きで書ける |
| 6 | 検証手順 | 未完了タスク各件に検証コマンド1行が書ける |
| 7 | 鮮度条件 | この文書がいつ stale になるか（expires_when）が書ける |
| 8 | 引き継ぎ先の想定 | 誰が（次の自分/別モデル/別人）どのタイミングで読むか |

未充足の観点だけ、vw-grill の wave 方式で質問する（AskUserQuestion、推奨回答付き、**最大2 wave**。
セッション終盤でユーザーは疲れている前提で、回答コストを最小にする）。
それでも埋まらない観点は「不明」として文書に残す（勝手に埋めない）。

## Phase 2: テンプレート生成

[handoff-template.md](./references/handoff-template.md) の固定セクションを**順番どおり全部**埋める。
保存先: `.brain/{project}/handover/{YYYY-MM-DD}-{slug}-HANDOFF.md`

生成中の必須手続き:

- **hash 分類**: 「完了」と書く各行で `jj log` / `git log --oneline -- <file>` を実行し hash を転記。見つからない → 「未確認」列へ
- **実出力転記**: 現状スナップショットの数値・状態は Phase 0 で保存したツール出力からコピーする（記憶から書かない）
- **二層構造**: 部分的に完了したタスクは「完了部分（hash 付き）/ 残り（Open）」に分割して両方書く

## Phase 3: 双方向突合 + 自己検証（提出前の必須手順）

1. **突合（順方向)**: 素材リスト（会話タスク・TaskList・jj log の各コミット）を上から走査し、HANDOFF のどのセクションに現れるか対応表を作る。現れないものは (a) 文書に追記する か (b) 「省いたもの」セクションに理由付きで載せる。**黙って落とすことは禁止**
2. **突合（逆方向)**: HANDOFF の全 claim を走査し、根拠（hash / ツール出力 / ユーザー発言）が随伴しているか確認。ないものは修正する
3. **精神論リント**: 「注意」「気をつけ」「意識」「しっかり」「適切に」で文書を検索し、ヒットした行を3点形式（発動条件→手順→根拠）に書き直す
4. 書き込み後 `ls -la <path> && wc -l <path>` を実行し、実出力を文書末尾の自己検証セクションに貼る

## Phase 4: 独立 verifier による採点（必須・スキップ不可）

self-critique は無効（実測で確認済み）。**fresh-context のサブエージェント**に採点させる:

1. [verifier-protocol.md](./references/verifier-protocol.md) の指示文テンプレートに、生成した HANDOFF のパスを埋めてサブエージェント（Sonnet 以上。**Haiku 不可** — 天井効果で欠陥を見逃す）を起動する
2. verifier は [rubric.md](./references/rubric.md) の5基準×5点で採点し、**採点手順に従って実ファイル照合**（hash の実在確認、検証コマンドの dry-run 等）を行う
3. **合格基準: 全基準 4点以上**。3点以下の基準があれば、verifier の指摘に従って修正 → 再採点（最大2周。2周で合格しなければ未達項目をユーザーに報告して判断を仰ぐ）
4. 最終スコアと verifier の指摘を文書の自己検証セクションに転記する

## Rollback / Recovery

- verifier が2周しても不合格: 未達基準と理由をユーザーに報告し、そのまま保存するか追加作業するかを確認する
- 旧 HANDOFF が同じトピックで存在する: 上書きせず新規日付で作成し、旧文書の frontmatter に `superseded_by:` を追記する（stale 文書の腐敗防止）
- 生成に失敗した場合: `git restore claude/.claude/skills/vw-handoff/` でスキル自体は復旧可能
