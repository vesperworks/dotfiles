<background>

## なぜこのスキルが存在するか

- **Karpathy LOOPS.md 原則VIII**: 「モデルが改善するたび、前四半期に書いたハーネスの半分はオーバーヘッドになる。各リリースごとに読み直し、モデルが無料でやるようになったものを削れ」
- **Anthropic 公式**: 「旧モデル向けの過剰に規範的なスキル・指示は上位モデルの品質をむしろ下げる。デフォルト性能が良ければ削除を検討せよ」
- **この環境の実証（2026-07-07 model-bench）**: 検証手順の列挙は Opus 4.8 の欠陥を 5→1 に減らしたが、Sonnet 5 には「列挙されたシナリオで十分」という天井を作り実問題の見逃しを生んだ。**同じルールがモデルによって薬にも毒にもなる**
- 詳細な実証データ: `.brain/dotfiles/handover/model-bench/results/` の REPORT.md / REPORT-AB.md

</background>

<workflow>

## Phase 0: スコープ確定と実測

1. 対象を列挙: グローバル CLAUDE.md / rules/*.md / プロジェクト CLAUDE.md / hooks/（強制実装側）
2. `wc -c` で常時注入サイズを実測し、前回レビュー時（後述の履歴ファイル）と比較
3. 前回からの増分 diff を `jj log` / `git log` で確認（何がいつ追加されたか）

## Phase 1: 3視点並列監査（サブエージェント3本）

[references/audit-prompts.md](./references/audit-prompts.md) のプロンプトを使い、3本を **1メッセージで並列起動**する:

| 監査 | 視点 | 検出対象 |
|------|------|---------|
| A | 矛盾・衝突 | ルール間矛盾 / ハーネス既定挙動（Auto mode 等）との綱引き / 適用範囲の誤配置 / サブエージェント波及不能 |
| B | 過剰規範性 | 裁量剥奪 / 天井効果 / オーバーヘッド強制 / 旧モデル前提の陳腐化 |
| C | 実害・実益突合 | hooks の偽陽性実績 / 二重管理箇所 / 実セッションでルールが働いた・邪魔した記録との照合 |

各監査の鉄則: **原文引用できる指摘のみ**（推測禁止）、深刻度 HIGH/MEDIUM/LOW、改善案1行。

## Phase 2:（任意）A/B 実測ベンチ

静的監査で HIGH 判定が出たルールについて、削除/変更の効果を実測したい場合のみ:
`.brain/dotfiles/handover/model-bench/README.md` のプロトコルを再利用（改訂前後の指示文 × 対象モデルで blind 実行 → 欠陥チェックリスト + 強制順位付け）。
静的監査だけで判断が明確な場合はスキップしてよい（コスト大）。

## Phase 3: 4分類判定とユーザー承認

全指摘を統合し、各ルールを4分類する:

| 分類 | 基準 | 例 |
|------|------|-----|
| **維持** | 実害を防いだ実証があり、モデルの自発性を縛らない | 実測転記ルール（fabricated report を実際に防いだ） |
| **行動文化** | 思想は正しいが原則名の列挙で終わっている → 1行の行動文に書き換え | 「YAGNI」→「頼まれていない機能を足すな」 |
| **スキル退避** | 常時注入が不要で、特定作業時だけ必要 → 該当スキル/プロジェクト CLAUDE.md へ移動 | bash 3.2 互換表（dotfiles 作業時のみ必要） |
| **削除** | 現行モデルが無指示でも同等以上にやる / 発火実績なし / 矛盾側が正 | 旧モデルの弱点前提の指示 |

判定に迷ったら: **「このルールを消したら、次のセッションで具体的に何が壊れるか」を1行で書けなければ削除候補**。

結果は `/html` status モード（各項目 diff つき、維持/変更/削除の3択）でユーザーに提示し、承認を得る。

## Phase 4: 適用と履歴記録

1. 承認された変更のみ適用（スキル本文の変更は最小 diff、[[feedback_dont_rewrite_skills_casually]] 遵守）
2. 変更後サイズを再実測し、`.brain/{project}/handover/harness-review-log.md` に追記:
   `日付 / 対象モデル世代 / 注入サイズ before→after / 変更件数（維持N・書換N・退避N・削除N）/ 主要判断の1行要約`
3. コミットは vw:commit で（ユーザー実行）

</workflow>

<constraints>

- **削除も変更もユーザー承認なしに実行しない**（Phase 3 の承認 UI を必ず挟む）
- 監査は原文引用ベースのみ。「〜かもしれない」だけの指摘は Phase 3 に含めない
- 実証済みルール（実害を防いだ記録があるもの）は、コスト面の指摘があっても「適用範囲の限定」までにとどめ、削除候補にしない
- ハーネスレビュー自体がルールを増やす結論に至った場合、増分は「削除された行数以下」を目安にする（単調増加の防止がこのスキルの存在理由）

</constraints>
